# `/VmAgent` (and `/Mongo`) API — swagger vs. hand-rolled types

Source: `docs/official-calculator-api-swagger.json` (OpenAPI 3, "Veeam Sizer Calculator API", `info.version: v1`, line 4 for title, line 5 for the full `info.description` — base path/auth/response-envelope notes below are transcribed from there).

## Summary

The swagger spec confirms the endpoint, method, and response envelope (`{success, data}` on 200, `POST https://calculator.veeam.com/vse/api/VmAgent`) that `docs/adr/0001-delegate-sizing-to-veeam-calculator-api.md` and the current proxy/client already assume. It also confirms `/Mongo` is byte-for-byte the same request/response contract as `/VmAgent` (swagger says so explicitly, line 1575's `summary`). Everything past that top-level shape in `src/types/vault-sizer-api.ts` diverges from the official schema in ways ranging from cosmetic (casing) to structural (a nested `retention` object that doesn't exist on this endpoint) to purely fictional (four request fields — `isCapTierVDCV`, `isManaged`, `machineType`, `instanceCount` — that appear nowhere in the entire 13,708-line spec, under any endpoint). The official `VmAgentInputs` schema has 46 properties; the current `VmAgentRequest` models 31, is missing several that materially change sizing behavior (`showPoints`, `growthFactor`, `projectLength`, `largeBlock`, `backupType`, `indexOptions`, etc.), and its nested retention shape is lifted from a schema (`Retention`/`GfsRetention`) that the spec documents as belonging to two entirely different endpoints (Public Cloud, DB Plugin), not VmAgent. The good news: the response side (`CRpsPoints` restore points, `TotalObjectTransactions`/`ObjectTransactions`) is a close match already, and the `diskPurpose` numeric codes in the current code's comment are wrong in one specific way worth fixing (see Discrepancy 6). No source code was modified — this is a read-only research note. The client/proxy also currently have no call site yet (`src/lib/api/vault-sizer-client.ts` is only exercised by its own test), so there's no production behavior at risk in fixing any of this now.

## Endpoint contract

- **Path / method**: `POST /VmAgent`, `operationId: VmAgent_CalculateStorage` (`docs/official-calculator-api-swagger.json:6432`, `:6438`).
- **Twin endpoint**: `POST /Mongo`, `operationId: Mongo_CalculateStorage` (`:1569`, `:1575`) — its `summary` states verbatim: "Functionally identical to `/VmAgent` — the same request (`VmAgentInputs`) and response (`CVmAgentReturnObject`) shapes, produced by the same underlying calculation. The route name is historical and unrelated to MongoDB." Its request/response `$ref`s point at the exact same `VmAgentInputs` / `CVmAgentReturnObjectApiResponse` schemas.
- **Servers** (`:8`–`:24`): `http://localhost:5015` (root-mounted, dev), `https://calculator.demo.cis.local/vse/api`, `https://calculator.staging.aws.cis.local/vse/api`, `https://calculator.veeam.com/vse/api` (Production). The `info.description` (`:5`) explains the base-path difference is `app.UsePathBase("/vse/api")` applied for any non-Development environment. `VAULT_SIZER_API_URL` in `functions/api/vault-sizer.ts:1` (`https://calculator.veeam.com/vse/api/VmAgent`) matches the documented Production server + path exactly.
- **Auth**: none. `info.description` (`:5`) states access is restricted only by a server-side CORS allowlist in `Program.cs`; there is no restriction on server-to-server calls. Consistent with the proxy not sending any credentials.
- **Request content types**: `application/json`, `text/json`, `application/*+json` (`:6438` region, `requestBody.content` keys). The proxy sends `Content-Type: application/json` — a supported type.
- **Response content types, 200**: `text/plain`, `application/json`, `text/json` all documented for the same `CVmAgentReturnObjectApiResponse` schema (likely a Swashbuckle artifact of not using `[Produces("application/json")]`, not a real behavioral quirk) — harmless for the proxy, which parses via `.json()` regardless of header.
- **Response envelope / errors**, per `info.description` (`:5`):
  - Success: HTTP 200, `{"success": true, "data": {...}}`.
  - Application-level error: HTTP 400, `{"success": false, "messages": [...]}` — this is the `ReturnMessage` schema (`:11814`): `success` (bool, always `false` here) + `messages` (nullable `string[]`), returned when "a calculation class throws or otherwise rejects the input."
  - Model-validation error: HTTP 400, a standard ASP.NET Core `ValidationProblemDetails` body — returned automatically when a request fails `[Range]`-based validation on `VmAgentInputs` _before_ the controller runs (e.g. `sourceTB` > 1,024,000, or `reduction` > 99.99 — see the per-field `maximum`/`minimum` constraints below). This shape is **not** in `components.schemas` (it's a framework standard type Swashbuckle doesn't materialize) — the spec explicitly warns "Both 400 shapes are possible for the same endpoint — callers should handle both `ReturnMessage` and `ValidationProblemDetails`."
  - Both documented 400 response schemas: `ReturnMessage` at `text/plain`, `application/json`, `text/json` (`:6438` region, `responses.400.content`).

### Nested: `IndexingInputObject` (`:10652`) — used by `VmAgentInputs.indexOptions`

- `enabled` (bool) — when false, `indexResults` in the response is fully zeroed; when true, `guestSourceFilesMil`/`emIndexRetentionMonths` drive a non-zero result.
- `guestSourceFilesMil` (int, 0–1,024,000) — millions of guest files per restore point (50MB uncompressed / 12.5MB compressed per million files).
- `emIndexRetentionMonths` (int, 0–1,024,000) — months of index data retained on the EM server.
- `emInstalledOnBackupServer` (bool) — for the **VmAgent** module specifically (per this property's own description), this only changes which of `indexResults.vbrServerSpaceGB` / `indexResults.emServerSpaceGB` is non-zero — both figures are always returned, unlike the BackupServer module which zeroes the whole object in one branch.
- `retentionPointsSumOfRegularAndGfs` (int) — documented as **inert in the VmAgent module**: "this property is not read at all — that module derives its own retention count from its own daily/weekly/monthly/yearly retention inputs."

### Confirmed NOT present anywhere in `VmAgentInputs` (or in the 107-schema document at all)

Checked by full-text search across the whole spec file — zero hits for any of: `isCapTierVDCV`, `isManaged`, `machineType`, `instanceCount`, `isGfsDefined`, `isDefined` (in a GFS/retention sense). These four request fields and two flags exist only in the current TypeScript types, not in the official API.

## Response schema — `CVmAgentReturnObject` (`docs/official-calculator-api-swagger.json:9383`–`9473`)

Wrapped by `CVmAgentReturnObjectApiResponse` (`:9474`): `{ success: boolean, data: CVmAgentReturnObject | null }`. Its description (`:9474`) is worth noting verbatim: this wrapper type is **documentation-only** — controllers actually return an anonymous C# object at runtime (`Ok(new { success = true, data = result })`); the two aren't compiler-linked, so if a controller's anonymous object shape ever drifts, this schema has to be hand-updated to match or the docs go stale. It also confirms PascalCase C# properties serialize as **camelCase** on the wire (`AddControllers()`, default `System.Text.Json`, no custom naming policy) — i.e. the casing shown below is what actually appears in responses.

### Nested: `ProxyResult` (`:11301`) → `ComputeRequirement` (`:9731`) → `Volume` (`:13612`) / `Throughput` (`:12430`) / `Location` (`:10712`)

- `ProxyResult`: `{ compute: ComputeRequirement (nullable), siteId: string (nullable) }`. `siteId` description: "Left empty for VM Agent."
- `ComputeRequirement`: `{ name: string (nullable), site: Location (nullable), cores: number, ram: number, volumes: Volume[] (nullable), networkThroughput: Throughput (nullable) }`. Cores/RAM are rounded up in Simple mode, raw in Advanced.
- `Volume`: `{ diskGB: number, throughputMbps: Throughput (nullable), diskPurpose: DiskType enum }`.
- `Throughput`: `{ inboundMBps: number, outboundMBps: number }` (`:12430`).
- `Location`: `{ siteId: string (nullable), siteName: string (nullable) }` (`:10712`).
- **`DiskType` enum** (`:10114`), full set — `diskPurpose` values: `0`=OS, `1`=OSStorage, `2`=RepoLocal, `3`=RepoObject ("Object storage repository"), `4`=RepoArchive ("Archive object repository"), `5`=Cache, `6`=VbrServerIndexStorageGB, `7`=EmServerIndexStorageGB, `8`=ProductInstallationSpaceGB, `9`=LogStorageGB, `10`=ConfigDatabaseStorageGB, `11`=DatastoreSpaceGB, `12`=RestorePointStorageGB, `13`=RepoCapacityTier ("Capacity Tier Object storage"), `30`=PremiumSsdDisk. Confirmed in the endpoint's own 200 response example (`responses.200.content["text/plain"].example`, in the `/VmAgent` path object): `repoCompute.compute.volumes` returns three entries with `diskPurpose` `3`, `13`, `4` in that order — i.e. performance tier (RepoObject), capacity tier (RepoCapacityTier), archive tier (RepoArchive).

### Nested: `IndexingReturnObject` (`:10686`)

`{ compute: ComputeRequirement (nullable), vbrServerSpaceGB: number, emServerSpaceGB: number }`.

### Nested: `TotalObjectTransactions` (`:12447`) → `ObjectTransactions` (`:11021`)

`TotalObjectTransactions`: `{ performanceTierTransactions, capacityTierTransactions, archiveTierTransactions }`, each an `ObjectTransactions`.
`ObjectTransactions`: `{ firstMonthTransactions: number, secondMonthTransactions: number, finalMonthTransactions: number }` — "API PUT calls, in thousands," for the first/second/final month of the retention window.

(Note: a same-shaped-sounding but **different, unused-by-VmAgent** schema, `TransactionResult`, also exists in the document — annual not monthly, with immutability-adjusted variants (`firstImmutTrans` etc.) and an `offloadType`/`offloadMonth` pair. It is not referenced anywhere in `CVmAgentReturnObject`'s chain; don't confuse the two.)

### Nested: `CRpsPoints` (`:8908`)

`{ pointType: string (nullable), day: integer, backupCapacity: number, isFull: boolean, isGFS: boolean, isImmutable: boolean, flags: string (nullable) }`. Description: "produced by the Restore Point Simulator (`CRpsSimulator`)" — `CRpsSimulator` is the internal C# class name; the schema/DTO itself is called `CRpsPoints`, not `CRpsSimulator`.

- `pointType` documented values: `"performanceTier"`, `"capacityTier"`, or `"archiveTier"` — all three, not just `"performanceTier"`. Confirmed in the endpoint's response example: entries with `pointType: "archiveTier"` and `"capacityTier"` both appear alongside `"performanceTier"`.
- `backupCapacity`: "Size of this specific restore point, in TB" — confirms the current code's "do not divide by 1024" comment (already-TB, not GB or bytes).
- `flags`: space-separated GFS slot tokens, e.g. `"W2 M1"`.

### `ReturnMessage` (`:11814`) — 400 error body (one of two possible shapes)

`{ success: boolean (always false here), messages: string[] (nullable) }`.

## Enums referenced by `VmAgentInputs`

- **`ProductVersion`** (`:11284`): `0` = `CurrentVersion` (v13: 20k workload limit, 8-core min, higher specs; on `BackupServer` also honors `DeploymentType`), `-1` = `PreviousVersion` (v12: 10k limit, 4-core min, ignores `DeploymentType`).
- **`CalculatorMode`** (`:9674`): `0` = `Simple` (floors to `CVariables` minimums), `1` = `Advanced` (raw values).
- **`HyperVisor`** (`:10626`): `0` Vmware, `1` HyperV, `2` Nutanix, `3` ProxMox, `4` Other. Nutanix/ProxMox get a lower per-core RAM assumption and a 6-core/6GB Simple-mode floor; the rest use the standard floor.
- **`BackupOptions`** (`:8430`): `0` = `Backup` (standalone job), `1` = `Copy` (backup copy job).
- **`DiskType`** (`:10114`): see table above under `Volume`.

No enum named `MachineType` exists anywhere in the document (`machineType` in the current code has no official counterpart at all — it isn't a misnamed version of `hyperVisor`, which is a separate, real field).

## Discrepancies vs. `src/types/vault-sizer-api.ts`

1. **Casing mismatches — code uses PascalCase for several fields the official schema serializes as camelCase.** `src/types/vault-sizer-api.ts:16` `ChangeRate` → official `changeRate` (`:13299`); `:17` `Reduction` → `reduction` (`:13306`); `:19` `GrowthRatePercent` → `growthRatePercent` (`:13270`); `:20` `GrowthRateScopeYears` → `growthRateScopeYears` (`:13277`); `:24` `Weeklies` → `weeklies` (`:13353`); `:25` `Monthlies` → `monthlies` (`:13360`); `:26` `Yearlies` → `yearlies` (`:13367`); `:28` `ObjectStorage` → `objectStorage` (`:13482`). By contrast, `sourceTB`, `backupWindowHours`, `blockGenerationDays`, `days`, `moveCapacityTierEnabled`, `capacityTierDays`, `copyCapacityTierEnabled`, `immutablePerf(Days)`, `immutableCap(Days)`, `archiveTier*`, `hyperVisor`, `calculatorMode`, `productVersion` are already camelCase and match. This is exactly the "inconsistent casing that looks like it was copied verbatim from observed JSON" the task description flagged — confirmed: the inconsistency is real, and the official casing is uniformly camelCase throughout.

2. **`Blockcloning` is both mis-cased and mis-spelled.** `src/types/vault-sizer-api.ts:27` `Blockcloning` vs. official `blockCloning` (`:13320`) — note the capital `C` in `Cloning` that the current code drops.

3. **The nested `retention` object doesn't exist on this endpoint — it's borrowed from a different endpoint's schema.** `src/types/vault-sizer-api.ts:8-12,22` define `RetentionPolicy { days, gfs: GfsPolicy { weeks, months, years }, isGfsDefined }` and use it as `VmAgentRequest.retention`. The official `VmAgentInputs` (`:13223`) has no `retention` property at all — GFS/retention counts are four **flat, top-level** fields: `days` (`:13313`), `weeklies` (`:13353`), `monthlies` (`:13360`), `yearlies` (`:13367`), confirmed by the endpoint's own request example (`days: 14, weeklies: 4, monthlies: 12, yearlies: 1`, all flat). The nested `Retention`/`GfsRetention` shape (`{days, gfs:{weeks,months,years}}`) **is** a real schema in the spec (`Retention` at `:11791`, `GfsRetention` at `:10466`) — but its own description says it's "shared by input DTOs across multiple product modules (currently the Public Cloud and Database Plugin calculators — see `PublicCloudInputs` and `DbPluginInputObject`)." Grepping every `$ref` to `#/components/schemas/Retention` in the file (lines 9914, 11398, 11407, 11438, 11447, 11473) confirms it's used only inside those two other endpoints' input schemas, never `VmAgentInputs`. It looks like whoever ported this type mixed up which endpoint's retention shape applies. Additionally, `isGfsDefined` and `GfsPolicy.isDefined` (`vault-sizer-api.ts:2,11`) have zero hits anywhere in the 13,708-line spec — they aren't a renamed/misplaced official field, they're invented.

4. **Four request fields have no official counterpart anywhere in the document.** `src/types/vault-sizer-api.ts:39` `isCapTierVDCV`, `:40` `isManaged`, `:41` `machineType`, `:45` `instanceCount` — confirmed via full-text search of the whole spec, zero matches for any of these four names under any of the 16 documented endpoints (`/BackupServer`, `/Cdp`, `/Db`, `/EntraId`, `/FileToTape`, `/Mongo`, `/One`, `/PublicCloud`, `/Securiti`, `/Tape`, `/Unstructured`, `/VB365`, `/Vbsf`, `/VmAgent`, `/VmReplication`). `machineType` in particular is not a misnamed `hyperVisor` — `hyperVisor` (`:13284`) is a separate, real, already-correctly-modeled field.

5. **31 of 46 official `VmAgentInputs` properties are modeled; the 15 missing include several that change sizing output, not just labels.** Missing and sizing-relevant: `showPoints` (`:13331` — gates whether `restorePoints` comes back populated at all; the response type already models `restorePoints?`, but nothing in the request can currently ask for it), `growthFactor` (`:13335`), `projectLength` (`:13346`), `largeBlock` (`:13342`), `backupType` (`:13404`), `copiesEnabled` (`:13412`), `storageType` (`:13445`), `enableEntropyScanning` (`:13486`), `preferPhysicalProxySizing` (`:13490`), `indexOptions` (`:13494`). Missing and label-only (not echoed, no sizing effect per their descriptions): `siteId`/`siteName`/`id`/`name` (`:13242`-`13263`), `workloadName`/`workloadId`/`repoId`/`repoName`/`capTierRepoId`/`archiveTierRepoId` (`:13452`-`13477`).

6. **`ComputeVolume.diskPurpose` comment is wrong for one of its three documented codes.** `src/types/vault-sizer-api.ts:50` comments `diskPurpose: number; // 3=perf, 13=capacity/cache, 4=logs`. Per the official `DiskType` enum (`:10114`): `3` = `RepoObject` ("Object storage repository"), `13` = `RepoCapacityTier` ("Capacity Tier Object storage" — not "cache"; `Cache` is a separate code, `5`), and **`4` = `RepoArchive`** ("Archive object repository") — **not** "logs" (`LogStorageGB` is code `9`, a different value entirely). The endpoint's own 200-response example confirms this ordering: `repoCompute.compute.volumes` returns exactly three entries with `diskPurpose` `3`, `13`, `4`, matching performance/capacity/archive tier respectively — the current comment's "4=logs" would misidentify the archive-tier volume as a logs volume.

7. **`ComputeSpec`/`ComputeNode` omit real fields from `ComputeRequirement`/`ProxyResult`.** `src/types/vault-sizer-api.ts:48-61` model only `{cores, ram, volumes}` inside `{compute}`. The official `ComputeRequirement` (`:9731`) also has `name` (string, nullable), `site` (`Location`, nullable), and — most notably — `networkThroughput` (`Throughput`: `inboundMBps`/`outboundMBps`), which is present in the endpoint's own example response for both `proxyCompute` and `repoCompute`. `Volume` (`:13612`) also has a per-volume `throughputMbps` (`Throughput`) that `ComputeVolume` (`vault-sizer-api.ts:48-51`) doesn't model. And `ProxyResult` (`:11301`) itself has a `siteId` sibling to `compute` that `ComputeNode` (`vault-sizer-api.ts:59-61`) doesn't model (documented as always empty for VmAgent, so low-value, but present on the wire).

8. **`id`/`siteId` are typed too loosely.** `src/types/vault-sizer-api.ts:102,104` type `id`/`siteId` as `number | string`. Official `CVmAgentReturnObject.id`/`.siteId` (`:9386`, `:9396`) are both `string, nullable` only — never numeric. Harmless in practice (TS union is a superset) but not accurate.

9. **`indexResults` is typed `unknown`; the schema fully describes it.** `src/types/vault-sizer-api.ts:106` types `indexResults?: unknown`. The official shape is `IndexingReturnObject` (`:10686`): `{ compute: ComputeRequirement | null, vbrServerSpaceGB: number, emServerSpaceGB: number }` (`:9429` for where it's referenced from `CVmAgentReturnObject`). This is consistent with discrepancy 5 — the request has no `indexOptions` to populate this meaningfully yet either.

10. **Response-side accuracy, for contrast — these already match well.** `RestorePoint` (`vault-sizer-api.ts:75-91`) matches `CRpsPoints` (`:8908`) field-for-field including casing (`pointType`, `day`, `backupCapacity`, `isFull`, `isGFS`, `isImmutable`, `flags`), modulo one stale comment: `:76`'s "Observed: 'performanceTier'. Capacity/archive tier values may appear in future responses" undersells it — the spec documents and the example response demonstrates all three values (`performanceTier`, `capacityTier`, `archiveTier`) today, not just as a future possibility. `MonthlyTransactions`/`TransactionCosts` (`vault-sizer-api.ts:63-73`) also match `ObjectTransactions`/`TotalObjectTransactions` (`:11021`, `:12447`) exactly, including casing. The top-level `VmAgentResponse { success, data }` wrapper (`vault-sizer-api.ts:110-113`) matches `CVmAgentReturnObjectApiResponse` (`:9474`) exactly.

11. **Proxy/client don't surface the two documented 400 error shapes.** `functions/api/vault-sizer.ts:39` forwards the upstream status and raw JSON body verbatim (`jsonResponse(data, upstream.ok ? 200 : upstream.status)`) — that part is fine and preserves whichever error shape the upstream sent. But `src/lib/api/vault-sizer-client.ts:14-18` throws a generic `Vault Sizer API error: ${status} ${statusText}` on any non-ok response without reading the body at all, discarding both possible documented 400 shapes: `ReturnMessage.messages` (`:11814`, human-readable validation/exception text) and the field-level `ValidationProblemDetails` errors from `[Range]` constraint violations (e.g. `sourceTB` > 1,024,000 at `:13292`, `reduction` > 99.99 at `:13306`). Since there's no call site yet (`vault-sizer-client.ts` is exercised only by its own test, `vault-sizer-client.test.ts`), this is a clean, low-risk place to fix before any UI wires up to it.

## Open questions

- **Would the current (wrong) request casing actually fail against the live API, or silently no-op?** The spec doesn't state whether the server's JSON deserialization is case-sensitive. ASP.NET Core Web API commonly defaults to case-insensitive property binding, which — if true here — would mean the current PascalCase fields (`ChangeRate`, `Weeklies`, etc.) still bind correctly to the same camelCase-serializing C# properties, and only the _structural_ issues (discrepancy 3: the nested `retention` object, which has no flat-field equivalent path to bind through) would actually be lost server-side. This spec provides no evidence either way — would need to be tested against the live endpoint (or the `vdc-vault-readiness` sibling project's actual request logs, if any exist) to confirm.
- **Does the server reject unknown top-level properties, or silently ignore them?** `additionalProperties: false` on `VmAgentInputs` (`:13223`) is a Swashbuckle schema annotation, not necessarily an enforced runtime contract — depends on `JsonSerializerOptions.UnmappedMemberHandling`, which the spec doesn't document. This affects how much discrepancies 3 and 4 (the fictional `retention`/`isCapTierVDCV`/`isManaged`/`machineType`/`instanceCount` fields) actually matter in practice versus being harmlessly dropped.
- **What does `ValidationProblemDetails` actually look like from this API?** It's a standard ASP.NET Core type but isn't materialized in `components.schemas`, so its exact field names (`errors`, `title`, `status`, etc.) aren't pinned down by this spec — would need a live 400 response with an out-of-range value to confirm the concrete shape.
- **Interaction between `storageType: "object"` and `objectStorage: true`.** Both are documented as independently able to trigger the same object-storage immutability-tax sizing path (`:13445`, `:13482`), but the spec doesn't say what happens if they disagree (e.g. `storageType` unset/non-"object" but `objectStorage: true`, or vice versa) — worth clarifying before modeling both in the request type.
